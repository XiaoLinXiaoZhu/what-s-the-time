# 专家级提示词设计能力框架

> **生成日期**: 2025-11-17  
> **目标**: 为多步骤提示词系统设计提供专家级能力框架  
> **应用**: 游戏剧情脚本转换系统

---

## 能力一：提示词系统架构设计

### A. 核心哲学
提示词系统架构设计的本质是**将混沌的创意需求转化为可验证、可迭代的确定性流程**。这不是简单的步骤堆砌，而是建立一个能够自我进化、质量可控的"创意生产系统"，让每个中间产物都成为可检查、可优化的独立单元。

### B. 第一原则

**原则1: 单一职责原则 (Single Responsibility Principle)**
每个提示词只负责一个明确的转换或验证任务。一个提示词不应该同时承担"理解需求"和"生成代码"两个职责，而应该专注于"将自然语言需求转换为结构化规格"或"验证代码是否符合规范"。这确保了每个步骤的输出都是高内聚、低耦合的，便于调试和优化。

**原则2: 文档即接口原则 (Document-as-Interface Principle)**
不同步骤之间通过标准化的文档格式进行通信，而不是通过内存或隐式约定。每个步骤的输入是前一个步骤输出的文档，输出是下一个步骤需要的文档。这使得整个系统可以被暂停、检查、回滚和并行化。

**原则3: 可验证性原则 (Verifiability Principle)**
每个中间步骤必须产生可快速验证的产物。验证不应该依赖主观判断，而应该基于明确的检查清单、模式匹配或自动化测试。这确保了系统质量可以在每个节点被确认，而不是等到最后才发现问题。

**原则4: 降噪迭代原则 (Denoising Iteration Principle)**
系统应该像扩散模型一样工作：从高度抽象、充满噪声的初始概念开始，通过多个专业化的步骤逐步去除不确定性、添加细节、优化质量。每个步骤都是一次"降噪"过程，让输出越来越接近最终目标。

### C. 可执行模型与框架

**框架1: PDCA循环嵌入模型 (Plan-Do-Check-Act Embedding)**
在每个提示词步骤中嵌入微型PDCA循环：
- **Plan**: 明确本步骤的输入文档格式、处理目标、输出文档格式
- **Do**: 执行转换或验证
- **Check**: 生成自检报告，说明处理了多少元素、发现了什么问题、做出了什么决策
- **Act**: 将问题和改进建议记录到元数据，供后续步骤或下一轮迭代参考

**框架2: 三阶验证模型 (Three-Tier Validation Model)**
- **Tier 1 - 语法验证**: 检查输出是否符合文档格式规范（如DSL标签是否闭合、JSON是否有效）
- **Tier 2 - 语义验证**: 检查内容是否符合领域规则（如游戏脚本的时间格式是否正确、角色对话是否符合人设）
- **Tier 3 - 意图验证**: 检查输出是否实现了原始需求的目标（如是否成功营造了预期的情感氛围）

**框架3: 专家链路由模型 (Expert Chain Routing Model)**
根据输入文档的特征动态路由到不同的专家提示词：
- 如果输入是"概念草案" → 路由到"概念整理专家"
- 如果输入是"时间线混乱" → 路由到"时间线校验专家"
- 如果输入是"细节不足" → 路由到"细节丰富专家"
路由决策本身也是一个可验证的步骤，生成路由理由文档。

### D. 高级技术与细节

**技术：元认知提示词 (Metacognitive Prompting)**
在每个专家提示词中要求其不仅生成转换结果，还要生成"处理日志"，包括：
- 我识别出了哪些关键元素？
- 我发现了哪些潜在问题？
- 我做出了什么权衡决策？
- 我对哪些部分不确定，需要后续验证？

这使得整个系统具有自我反思能力，后续步骤可以基于这些元认知信息做出更智能的决策。例如，如果"概念整理专家"在日志中表示"对角色动机理解不确定"，那么"细节丰富专家"就会特别关注这个部分，而不是盲目添加无关细节。

**技术：对抗性验证 (Adversarial Validation)**
引入专门的"批评家"提示词，其唯一职责是找出前一个专家输出的问题。这个批评家不知道前一个专家的意图，只根据文档规范和领域知识进行攻击。这种对抗性机制能够发现单一专家视角的盲点，大幅提升质量。

### E. 关键反模式

**反模式1: 步骤过度耦合 (Over-coupling Steps)**
症状：一个步骤的输出格式依赖于另一个步骤的内部实现细节，修改一个步骤需要同时修改多个步骤。
后果：系统变得脆弱，无法独立优化单个步骤。
避免方法：严格定义文档接口，使用版本控制，确保向后兼容。

**反模式2: 验证后置 (Late Validation)**
症状：所有步骤完成后才进行一次性验证，中间产物无法检查。
后果：问题发现太晚，修复成本极高，可能需要重做整个流程。
避免方法：在每个步骤后嵌入验证检查点，生成验证报告。

**反模式3: 专家角色模糊 (Vague Expert Roles)**
症状：多个提示词的职责边界不清，都能做相似的事情。
后果：系统行为不确定，难以调试，质量不稳定。
避免方法：为每个专家提示词编写"职责宪章"，明确输入、输出、核心能力和不做什么。

**反模式4: 过度工程化 (Over-engineering)**
症状：为每个可能的边缘情况都设计专门步骤，系统复杂度过高。
后果：维护成本超过收益，简单的任务也需要经过过多步骤。
避免方法：遵循"先简单，后复杂"原则，只有当某个问题反复出现且影响质量时才引入新步骤。

## 能力二：文档驱动开发

### A. 核心哲学
文档驱动开发的本质是**将代码视为文档的附属产物，而非相反**。在提示词系统中，文档不仅是沟通的媒介，更是系统状态的唯一真实来源。每个文档都是一个"检查点"，可以被版本控制、审查、测试和复用。这种思维方式将开发从"写代码"转变为"设计文档转换流程"。

### B. 第一原则

**原则1: 文档即代码原则 (Document-as-Code Principle)**
所有中间文档都应该：
- 使用结构化格式（Markdown、YAML、JSON、DSL）
- 受版本控制
- 可以通过自动化工具验证
- 包含元数据（作者、版本、生成时间、依赖关系）
- 可以被diff和merge

这使得文档具有代码的所有优点：可追溯、可协作、可测试、可部署。

**原则2: 单源真理原则 (Single Source of Truth)**
任何信息只应该在文档中存在一次，其他地方的相同信息都应该是引用或派生。例如，游戏角色的性格描述只应该在"角色设定文档"中详细定义，其他文档（如剧情脚本、提示词）都应该引用这个源头。这确保了信息的一致性，修改一处即可全局生效。

**原则3: 文档自描述原则 (Self-Describing Document Principle)**
每个文档应该包含足够的信息，让一个不了解背景的人也能理解其用途和结构。这包括：
- 文档标题和用途说明
- 格式规范引用
- 示例和模板
- 验证规则
- 与其他文档的关系

这使得系统具有自我文档化能力，新成员可以快速上手。

**原则4: 转换可追溯原则 (Traceability Principle)**
每个文档的生成过程应该被记录，包括：
- 由哪个提示词生成
- 输入了哪些文档
- 做出了什么决策
- 有哪些备选方案被考虑

这使得整个系统可以被审计，当出现问题时可以回溯到具体步骤。

### C. 可执行模型与框架

**框架1: 文档类型系统 (Document Type System)**
为每种文档定义严格的类型规范：
```typescript
interface ConceptDraft {
  type: 'concept_draft';
  version: '1.0';
  metadata: {
    source: string;
    generatedBy: string;
    timestamp: string;
  };
  content: {
    coreIdea: string;
    themes: string[];
    characters: CharacterOutline[];
    timeline?: TimelineEvent[];
  };
  validation: {
    requiredFields: string[];
    constraints: ValidationRule[];
  };
}
```

**框架2: 文档依赖图 (Document Dependency Graph)**
使用有向无环图（DAG）建模文档之间的依赖关系：
- 节点：文档
- 边：转换关系
- 权重：转换复杂度、置信度

这使得可以：
- 并行化独立转换
- 识别关键路径
- 评估修改影响范围
- 优化转换顺序

**框架3: 文档差异分析模型 (Document Diff Analysis Model)**
不仅比较文本差异，还要比较语义差异：
- 结构差异：章节、段落、列表的变化
- 语义差异：概念、关系、约束的变化
- 影响分析：哪些下游文档需要重新生成

这使得可以智能地决定哪些步骤需要重新执行，而不是全量重算。

### D. 高级技术与细节

**技术：活文档 (Living Documentation)**
文档不是静态的，而是与系统同步演化的：
- 当代码变更时，相关文档自动标记为"待更新"
- 当文档更新时，依赖它的提示词自动重新执行
- 文档中包含动态部分，可以嵌入查询结果或统计数据

例如，"角色设定文档"中可以嵌入一个动态表格，显示该角色在所有剧情片段中出现的次数和情感变化趋势，这些数据是实时从剧本中统计的。

**技术：文档模板引擎 (Document Template Engine)**
为每种类型的文档创建智能模板：
- 包含必填字段和可选字段
- 提供示例和提示
- 嵌入验证规则
- 支持条件渲染（根据内容类型显示不同章节）

这使得创建新文档时，作者会得到智能引导，减少格式错误。

**技术：文档质量门禁 (Document Quality Gates)**
在文档进入下一个步骤之前，必须通过自动化检查：
- 格式验证：是否符合Schema
- 完整性检查：必填字段是否都有值
- 一致性检查：与其他文档是否有冲突
- 质量评分：基于启发式规则给出质量分数

未通过门禁的文档会被退回，并生成详细的改进建议。

### E. 关键反模式

**反模式1: 文档冗余 (Document Redundancy)**
症状：相同信息在多个文档中重复出现，修改时需要同步更新多处。
后果：信息不一致，维护成本高，容易出错。
避免方法：建立文档引用机制，使用链接或ID引用，而不是复制内容。

**反模式2: 文档僵尸 (Document Zombies)**
症状：文档存在但无人维护，内容过时但仍在被引用。
后果：系统基于错误信息运行，产生低质量输出。
避免方法：实施文档生命周期管理，定期审查，自动标记长时间未更新的文档。

**反模式3: 过度结构化 (Over-structuring)**
症状：为了追求完美的结构，创建了大量嵌套和复杂的文档格式。
后果：文档难以编写和阅读，工具开发成本高，灵活性差。
避免方法：遵循"足够结构"原则，只在能带来明确价值的地方引入结构。

**反模式4: 文档与实现脱节 (Document-Implementation Gap)**
症状：文档描述的是理想状态，但实际实现已经不同步。
后果：文档失去可信度，团队不再依赖文档。
避免方法：将文档验证纳入CI/CD流程，确保文档与实现同步更新。

## 能力三：降噪迭代思维

### A. 核心哲学
降噪迭代思维的本质是**将创作过程视为从噪声到信号的渐进式提取**。初始的概念草案充满不确定性、矛盾和不完整，就像充满噪声的信号。每个迭代步骤都是一个专门的滤波器，去除一类特定的噪声（逻辑矛盾、细节不足、情感不连贯等），同时保留和增强有效信息。这个过程不是随机的试错，而是有理论指导的、可预测的渐进优化。

### B. 第一原则

**原则1: 噪声分类原则 (Noise Classification Principle)**
不同类型的噪声需要不同的处理策略：
- **逻辑噪声**：概念矛盾、时间线错误、因果关系混乱 → 需要逻辑校验步骤
- **细节噪声**：描述模糊、感官信息不足、环境描写缺失 → 需要细节丰富步骤
- **情感噪声**：情感转折突兀、动机不合理、氛围不一致 → 需要情感调优步骤
- **技术噪声**：格式错误、标签未闭合、接口不匹配 → 需要技术校验步骤

只有先分类，才能选择正确的降噪算法。

**原则2: 保真度优先原则 (Fidelity-First Principle)**
在降噪过程中，必须优先保证不丢失有效信息。一个保守的降噪（去除少量噪声，保留所有信号）优于激进的降噪（去除大量噪声，但可能丢失信号）。这要求每个步骤都有"信息保留率"指标，确保有效信息不被意外过滤。

**原则3: 渐进细化原则 (Progressive Refinement Principle)**
降噪应该按照从粗到精的顺序进行：
1. 先处理宏观结构（整体框架、主要情节）
2. 再处理中观逻辑（因果关系、角色行为）
3. 最后处理微观细节（语言表达、节奏控制）

过早关注细节会导致大量返工，因为宏观调整会摧毁已完成的细节工作。

**原则4: 反馈驱动原则 (Feedback-Driven Principle)**
每个降噪步骤都应该产生可度量的反馈，指导下一步：
- 本次降噪去除了多少矛盾？
- 增加了多少有效信息？
- 哪些部分仍然不确定？
- 哪些部分需要特别关注？

这使得迭代过程可以被数据驱动，而不是凭感觉。

### C. 可执行模型与框架

**框架1: 四阶段降噪模型 (Four-Stage Denoising Model)**
```
阶段1: 概念整理 (Concept Organization)
- 输入: 混乱的创意草案
- 降噪目标: 去除逻辑矛盾，建立基本结构
- 输出: 结构化的概念文档
- 验证: 概念一致性检查

阶段2: 时间线校验 (Timeline Validation)
- 输入: 结构化概念文档
- 降噪目标: 去除时间矛盾，确保因果合理
- 输出: 时间线一致的剧情大纲
- 验证: 时间逻辑检查

阶段3: 细节丰富 (Detail Enrichment)
- 输入: 剧情大纲
- 降噪目标: 去除细节不足，增强沉浸感
- 输出: 细节丰富的场景描述
- 验证: 感官信息密度检查

阶段4: 技术转换 (Technical Conversion)
- 输入: 丰富的场景描述
- 降噪目标: 去除技术不规范，转换为可执行格式
- 输出: 符合规范的程序脚本
- 验证: 语法和接口检查
```

**框架2: 信噪比度量模型 (Signal-to-Noise Ratio Metrics)**
为每个文档定义可计算的指标：
- **信息熵**: 衡量文档的不确定性程度
- **矛盾密度**: 每千字的逻辑矛盾数量
- **细节完整度**: 必需细节字段的填充率
- **情感连贯性**: 相邻段落的情感转移合理性

这些指标使得降噪效果可以被量化比较。

**框架3: 自适应迭代深度模型 (Adaptive Iteration Depth Model)**
不是所有部分都需要相同程度的迭代：
- 核心情节: 深度迭代（多次校验和优化）
- 次要情节: 中度迭代（基本校验）
- 背景信息: 轻度迭代（格式检查）

系统根据内容的重要性自动分配迭代资源，避免过度工程化。

### D. 高级技术与细节

**技术：矛盾熵检测 (Contradiction Entropy Detection)**
不仅检测显式矛盾（如时间线错误），还检测隐式矛盾：
- 角色行为与性格设定不符
- 情感强度与事件重要性不匹配
- 环境描写与氛围基调冲突

通过计算"矛盾熵"来量化整体一致性，指导降噪优先级。

**技术：细节重要性预测 (Detail Importance Prediction)**
使用启发式规则预测哪些细节对沉浸感最重要：
- 首次出现的场景: 需要详细的环境描写
- 情感转折点: 需要丰富的心理描写
- 关键物品: 需要多感官的细节
- 重复场景: 只需变化部分的细节

这使得细节丰富步骤可以聚焦于高价值区域。

**技术：情感梯度平滑 (Emotional Gradient Smoothing)**
分析情感变化的陡峭程度，自动插入过渡：
- 如果情感从"平静"直接跳到"暴怒"，添加"不安→烦躁→愤怒"的中间状态
- 如果动机从"好奇"直接变为"恐惧"，添加"疑虑→警惕→害怕"的过渡

这使得情感转折更加自然，避免突兀。

### E. 关键反模式

**反模式1: 过度降噪 (Over-denoising)**
症状：为了追求完美的逻辑和一致性，去除了所有模糊性和不确定性。
后果：作品失去神秘感和想象空间，变得平淡无奇。
避免方法：保留可控的模糊性，特别是在艺术作品中，不确定性往往是魅力的来源。

**反模式2: 降噪顺序错误 (Wrong Denoising Order)**
症状：先优化语言表达，再调整宏观结构，导致大量细节工作被浪费。
后果：效率低下，团队士气受挫。
避免方法：严格遵守"宏观→中观→微观"的降噪顺序。

**反模式3: 缺乏降噪度量 (Lack of Denoising Metrics)**
症状：凭感觉判断降噪效果，无法量化改进。
后果：无法确定迭代是否收敛，可能陷入无限优化。
避免方法：建立信噪比指标体系，设定收敛阈值。

**反模式4: 忽视保真度 (Ignoring Fidelity)**
症状：为了去除噪声，过度修改原始创意，丢失了核心灵感。
后果：最终产物与原始愿景偏离，创作者失去认同感。
避免方法：建立"创意DNA"检查点，确保核心元素不被修改。

## 能力框架应用指南

### 如何选择和组合能力

**场景1: 从零构建提示词系统**
- 主要能力: 提示词系统架构设计
- 辅助能力: 文档驱动开发
- 应用方式: 先设计整体架构，再定义文档规范

**场景2: 优化现有提示词流程**
- 主要能力: 降噪迭代思维
- 辅助能力: 文档驱动开发
- 应用方式: 分析现有流程的噪声点，引入度量和反馈

**场景3: 团队协作的提示词开发**
- 主要能力: 文档驱动开发
- 辅助能力: 提示词系统架构设计
- 应用方式: 建立文档规范和质量门禁，确保协作效率

### 实施建议

1. **从小开始**: 不要一次性实施所有能力，选择一个最痛点开始
2. **度量驱动**: 为每个能力实施建立度量指标，验证效果
3. **持续优化**: 定期回顾能力应用效果，调整策略
4. **知识沉淀**: 将实践经验沉淀为团队规范，形成组织能力
