# STAGE 2: ANALYZE - 系统性根因分析

## 一、问题陈述

用户需要设计一个**多步骤提示词系统**，将游戏的基础概念和设计草案转换为游戏剧情脚本（TypeScript格式）。当前缺少一个完整的背景信息文档，无法向另一个 agent 清晰地传达项目需求。

## 二、五问法分析（Five Whys）

### Why-1: 为什么需要整理背景信息？
**答案**：用户需要向另一个 agent 提出要求，设计多步骤提示词系统。另一个 agent 需要理解项目背景才能设计出合适的提示词。

### Why-2: 为什么另一个 agent 需要理解项目背景？
**答案**：多步骤提示词系统需要：
- 理解输入格式（游戏设计草案）
- 理解输出格式（TypeScript脚本）
- 理解中间转换流程（编剧创作 → 技术转换）
- 理解质量要求（文风、叙事诡计、技术规范）

### Why-3: 为什么需要多步骤提示词系统？
**答案**：从游戏设计到最终脚本的转换是一个复杂的创作过程，需要：
- 知识整理（理解游戏设计、人物设定、时间线等）
- 结构规划（确定片段结构、时间锚点、解锁条件等）
- 内容创作（编写符合文风要求的文本）
- 格式转换（将自然文本转换为技术格式）
- 质量校验（检查文风、叙事诡计、技术规范等）

### Why-4: 为什么需要分步骤执行？
**答案**：单一提示词无法同时完成所有任务，因为：
- **职责冲突**：知识整理、创作、转换、校验需要不同的专业能力
- **信息过载**：一次性处理所有信息会导致质量下降
- **迭代优化**：需要中间状态文档进行校验和优化
- **可控性**：分步骤可以更好地控制每个环节的质量

### Why-5（系统根因）：为什么当前缺少系统化的转换流程？
**答案（系统根因）**：**项目缺少一个系统化的、可复用的创作工作流设计**。当前虽然有：
- 编剧创作指南（告诉编剧怎么写）
- 技术转换指南（告诉技术美术怎么转）
- 格式规范（定义输入输出格式）

但缺少：
- **从游戏设计到编剧创作的转换流程**
- **多步骤提示词系统的设计方法论**
- **中间状态文档的规范和校验机制**
- **质量保证和迭代优化的机制**

## 三、分层假设分析

### 3.1 局部层面（Local）
**假设**：用户只是需要整理背景信息，创建一个 `background.md` 文件。

**验证**：❌ 不充分。虽然这是当前任务，但用户明确提到需要设计"多步骤提示词系统"，这需要更深层的系统设计。

### 3.2 组件层面（Component）
**假设**：问题在于缺少一个完整的背景信息文档，导致另一个 agent 无法理解项目。

**验证**：⚠️ 部分正确。这确实是当前需要解决的问题，但不是根本问题。即使有了背景信息文档，如果缺少系统化的转换流程设计，另一个 agent 仍然无法设计出高质量的多步骤提示词系统。

### 3.3 系统层面（Systemic）
**假设**：**项目缺少一个系统化的创作工作流设计方法论**。这个方法论应该包括：
1. **输入输出规范**：明确定义每个步骤的输入和输出格式
2. **职责划分**：明确每个步骤的单一职责和专业能力要求
3. **中间状态文档规范**：定义中间状态文档的格式和校验标准
4. **质量保证机制**：定义每个步骤的质量检查点和迭代优化机制
5. **提示词设计模板**：基于《提示词优化方法.md》和《DSL规范设计.md》，为每个步骤设计标准化的提示词模板

**验证**：✅ 正确。这是系统根因。

## 四、系统根因总结

### 4.1 核心问题
**项目缺少一个系统化的、可复用的创作工作流设计方法论**，无法支持从游戏设计到最终脚本的系统化转换。

### 4.2 具体表现
1. **流程不明确**：从游戏设计到编剧创作，再到技术转换的流程不够清晰
2. **职责不清晰**：每个步骤应该由谁（或哪个 agent）负责，职责边界不明确
3. **质量标准缺失**：缺少中间状态文档的质量检查标准
4. **迭代机制缺失**：缺少基于中间状态文档的迭代优化机制
5. **提示词模板缺失**：缺少基于项目规范的标准化提示词设计模板

### 4.3 影响范围
- **当前任务**：无法向另一个 agent 清晰地传达项目需求
- **未来扩展**：无法系统化地处理新的游戏设计需求
- **质量保证**：无法系统化地保证每个步骤的输出质量
- **协作效率**：多个 agent 或人员协作时，缺少统一的工作流标准

## 五、解决方案方向

### 5.1 短期方案（Patch）
**创建 `background.md`**：整合所有必要的背景信息，让另一个 agent 能够理解项目。

**局限性**：
- 只能解决当前的信息传递问题
- 无法解决系统化的转换流程设计问题
- 另一个 agent 仍然需要自己设计多步骤提示词系统

### 5.2 长期方案（Refactor）
**设计系统化的创作工作流方法论**，包括：
1. **工作流设计框架**：定义从游戏设计到最终脚本的完整转换流程
2. **步骤职责规范**：明确每个步骤的输入、输出、职责和质量标准
3. **中间状态文档规范**：定义中间状态文档的格式和校验标准
4. **提示词设计模板**：基于项目规范，为每个步骤设计标准化的提示词模板
5. **质量保证机制**：定义每个步骤的质量检查点和迭代优化机制

**优势**：
- 可复用的方法论，适用于未来的项目
- 系统化的质量保证机制
- 清晰的职责划分，支持多 agent 协作
- 标准化的提示词模板，提高效率和质量

## 六、推荐方案

**强烈推荐采用长期方案（Refactor）**，原因：
1. **系统性解决**：不仅解决当前问题，还为未来提供可复用的方法论
2. **质量保证**：系统化的质量检查机制，确保每个步骤的输出质量
3. **可扩展性**：适用于新的游戏设计需求和项目扩展
4. **协作效率**：标准化的流程和模板，提高多 agent 协作效率

**实施建议**：
1. **第一阶段**：创建 `background.md`（解决当前需求）
2. **第二阶段**：设计系统化的创作工作流方法论（解决根本问题）
3. **第三阶段**：基于方法论，设计多步骤提示词系统（应用方法论）

## 七、关键洞察

### 7.1 问题本质
这不是一个简单的"整理背景信息"任务，而是一个**系统化的创作工作流设计**问题。

### 7.2 解决思路
应该采用**系统化设计思维**，不仅提供背景信息，还要提供：
- 工作流设计框架
- 步骤职责规范
- 中间状态文档规范
- 提示词设计模板
- 质量保证机制

### 7.3 价值创造
通过系统化的方法论设计，不仅解决当前问题，还为项目建立了一个**可复用的创作工作流标准**，这将大大提高未来项目的效率和质量。
