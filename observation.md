# STAGE 1: OBSERVE - 观察与事实收集

## 一、用户需求分析

### 1.1 核心需求
用户希望设计一个**多步骤提示词系统**，将游戏的基础概念和设计草案，逐步转换为类似 `P0-CORE-T1037-FallBlur.ts` 这样的游戏剧情脚本。

### 1.2 目标效果
1. **类似 diffusion 模型的降噪过程**：一步一步执行，从模糊到清晰
2. **单一职责原则**：每一步由单一职责的专家负责
3. **中间状态文档**：每一步产出 n 个可检查的中间状态文档
4. **文档交换**：不同步骤间使用文档进行信息交换

### 1.3 最终产出要求
- 最终的多步骤提示词，将优化、细化和校验步骤拆分
- 例如：knowledge_整理 -> schedule -> checker1 -> detailer2 -> etc
- 每一步提供若干个以往的文档交由给最专业的专家
- 聚焦执行最小迭代，完成一个可以快速进行校验的中间状态文档

### 1.4 当前任务
用户现在需要我帮助整理所有必要的背景信息到 `background.md` 中，以便向另一个 agent 提出上述要求。

## 二、项目背景信息

### 2.1 项目概述
- **项目名称**：What's the time
- **项目类型**：多周目文字冒险游戏
- **核心机制**：玩家输入时间（HH:MM），游戏据此播放对应的"记忆片段"
- **叙事风格**：外壳为轻松的校园恋爱喜剧，内核是预谋杀人、记忆扭曲与叙事诡计
- **最终体验目标**：类似于读完《人间失格》后，那种对人性深渊的凝视所带来的、无法言说且挥之不去的压抑与虚无感

### 2.2 项目结构
```

│   │   ├── 提示词优化方法.md      # 提示词设计指南

│   ├── segments/

src/data/script/segments/

```

### 2.3 关键文档内容

#### 2.3.1 提示词设计方法（提示词优化方法.md）
- **核心原则**：定义系统的"为什么"而非"做什么"
- **设计优先级**：核心目标 → 背景/上下文 → 硬性边界 → 软性边界
- **表达原则**：使用动机语言，保持概念一致性
- **结构设计**：遵循层级结构，避免行为清单

#### 2.3.2 DSL规范（DSL规范设计.md）
- **标记系统**：`<core_goal>`, `<context>`, `<hard_boundary>`, `<soft_boundary>`, `<scale>`, `<principle>`, `<example>`, `<note>`, `<index>`
- **语法规则**：XML闭合标签格式、索引机制、嵌套规则
- **使用指南**：何时使用DSL、如何使用DSL、常见错误

#### 2.3.3 中间格式规范（中间格式规范.md）
- **目标读者**：编剧、技术美术
- **格式结构**：YAML Front Matter（片段元数据）+ 自然小说格式文本
- **格式规范**：
  - 内心独白/旁白：`*斜体*`
  - 强调：`**粗体**`
  - 对话：自然格式 `角色名: "对话内容"`
  - 换行：自然换行

#### 2.3.4 技术转换指南（脚本技术转换指南.md）
- **目标读者**：技术美术
- **转换原则**：理解文风原则和创作意图，手动转换为程序格式
- **转换内容**：
  - 文本类型判断（narration vs dialogue）
  - 节奏与停顿处理（`{br}`, `{delay}`）
  - 视觉强调处理（`{red}`, `{bold}`, `{typewriter}`等）
  - 交互元素实现（choice, timeChoice, command等）

#### 2.3.5 编剧创作指南（编剧创作指南.md）
- **文风目标**：建立"人间失格"氛围
- **四大支柱**：
  1. 不可靠的第一人称
  2. 疏离感
  3. 耻感与自我厌恶
  4. 语言的"表演性"
- **表现技法**：反复/纠结/对比表现、比喻与意象策略、精简与节奏的平衡

#### 2.3.6 游戏设计文档（游戏设计.md）
- **核心真相**：两次杀人，同名受害者；栽赃与自我受害叙事；代偿人格与"她"的模糊身份；时间与记忆的扭曲；最终 meta 高潮
- **核心玩法**：自由输入 + 有限锚点；阶段和flag系统；普通时间点（一问一答）；重叠时间点（两问一锁）；记忆空白与模糊区间
- **叙事诡计**：代词模糊；时间不标日期；同名混淆；多周目记忆残留；现实时间突破

#### 2.3.7 时间线文档（时间线.md）
- **完整时间线**：从童年 → A 案前 → A 案 → A 案后一年 → B 案前 → B 案 → B 案后 → 精神循环/多周目 → Meta 突破
- **关键时间锚点**：03:15（童年记忆）、08:05（A案前正常早晨）、10:37（A案和B案）、11:00（问话场景）、18:40（B案当晚共犯）

#### 2.3.8 人物设定（人物设定.md）
- **核心人物**：主角（"我"）、A（旧案受害者）、B（现案受害者）、E（共犯）、H（替罪羊）、D（B的朋友）、F（老师）、G（警官）
- **人物关系**：复杂的人物关系网，涉及控制、替代、栽赃、共犯等主题

### 2.4 目标输出格式（P0-CORE-T1037-FallBlur.ts）
- **TypeScript 格式**：符合 `ScriptSegment` 接口
- **结构**：
  - 片段元数据（id, time, description, loop, unlockFlags）
  - lines 数组（包含 narration, dialogue, choice, command 等类型）
  - 文本格式标记（`{br}`, `{delay}`, `{red}`, `{bold}`, `{typewriter}`等）

### 2.5 转换流程
1. **编剧创作**：使用中间格式（Markdown + YAML Front Matter）
2. **技术美术转换**：将中间格式转换为 TypeScript 程序格式
3. **质量要求**：理解创作意图，手动转换，而非机械转换

## 三、关键约束与要求

### 3.1 提示词设计要求
- 必须遵循《提示词优化方法.md》中的原则
- 必须使用《DSL规范设计.md》中的DSL格式
- 必须定义"为什么"而非"做什么"
- 必须使用动机语言，避免行为清单

### 3.2 转换流程要求
- 编剧使用自然小说格式，不关心技术细节
- 技术美术理解创作意图，手动转换为程序格式
- 每一步都有明确的输入和输出格式

### 3.3 质量要求
- 文本质量必须是顶级的
- 高密度信息：每一个片段都不能是废话
- "Aha!"时刻的精心设计
- 对"错误"输入的反馈

## 四、观察到的系统性问题

### 4.1 信息分散
- 项目信息分散在多个文档中
- 没有统一的背景信息文档
- 新 agent 需要阅读大量文档才能理解项目

### 4.2 转换流程不明确
- 从游戏设计到最终脚本的转换流程不够明确
- 缺少多步骤提示词系统的设计
- 缺少中间状态文档的规范

### 4.3 质量保证机制缺失
- 缺少校验步骤的设计
- 缺少中间状态文档的检查标准
- 缺少迭代优化的机制
