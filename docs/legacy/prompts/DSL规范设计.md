# 提示词设计DSL规范

> **用途**：用于规范表达提示词设计的结构化语言  
> **目标**：既满足LLM解析需求，又保持人类可读性  
> **原则**：简洁、明确、结构化

---

## 一、基本语法

### 1.1 标记系统

| 标记 | 用途 | 必需性 |
|------|------|--------|
| `<core_goal>` | 核心目标/动机 | 必需 |
| `<context>` | 背景/上下文信息 | 可选 |
| `<hard_boundary>` | 硬边界（客观约束） | 可选 |
| `<soft_boundary>` | 软边界（行为倾向） | 可选 |
| `<scale>` | 量表定义 | 条件必需* |
| `<principle>` | 设计原则 | 可选 |
| `<example>` | 示例（正确/错误） | 可选 |
| `<note>` | 说明/注意事项 | 可选 |
| `<index>` | 索引/概览 | 可选但推荐 |

*条件必需：当需要定义模糊边界时使用

### 1.2 语法规则

#### 规则1：XML闭合标签格式
- **使用XML格式的闭合标签**：`<tag_name>...</tag_name>`
- 开始标签：`<tag_name>`
- 结束标签：`</tag_name>`
- 内容在标签之间，可以多行
- 内容使用自然语言，保持可读性
- **优势**：LLM可以明确识别每个片段的边界

#### 规则2：索引机制（Index）
- **推荐使用索引**：在详细内容之前，先提供索引/概览
- 索引格式：`<index>...</index>`
- 索引内容：列出该部分包含的主要要点（1-3个要点）
- **优势**：
  - 重复并强调关键信息
  - 帮助LLM建立文档网络的理解
  - 提供快速概览

#### 规则3：嵌套规则
- `<soft_boundary>` 可以包含 `<scale>`
- `<example>` 可以包含多个子示例
- `<principle>` 可以包含 `<example>`
- 嵌套时标签必须正确闭合
- 嵌套层级使用缩进表示

#### 规则4：可选/必需规则
- `<core_goal>` - **必需**：每个提示词设计必须定义核心目标
- `<context>` - **可选**：仅在需要提供上下文时使用
- `<hard_boundary>` - **可选**：仅在存在硬约束时使用
- `<soft_boundary>` - **可选**：仅在需要定义行为倾向时使用
- `<scale>` - **条件必需**：仅在需要定义模糊边界时使用
- `<index>` - **可选但推荐**：为复杂部分提供索引

## 二、标记语义定义

### 2.1 <core_goal>（核心目标）

**语义**：描述系统的核心目标、动机或意图

**要求**：
- 必须使用动机语言，而非行为语言
- 应该简洁明确，1-3句话
- 可以多行，但避免冗长
- **推荐使用索引**：如果内容较复杂，先提供索引

**示例（基础格式）**：
```xml
<core_goal>
系统致力于高效、准确地完成用户指定的任务。
系统重视任务完成的质量和用户体验。
</core_goal>
```

**示例（带索引）**：
```xml
<core_goal>
<index>
- 核心目标：高效、准确地完成任务
- 价值导向：重视质量和用户体验
</index>

系统致力于高效、准确地完成用户指定的任务。
系统重视任务完成的质量和用户体验。
系统会根据任务类型和用户需求，选择最合适的完成方式。
</core_goal>
```

**错误示例**：
```xml
<core_goal>
系统会执行以下操作：
1. 分析用户需求
2. 生成代码
3. 检查代码质量
</core_goal>
```
❌ 问题：使用行为语言，而非动机语言

### 2.2 <context>（背景/上下文）

**语义**：提供系统运行的环境、角色、场景等背景信息

**要求**：
- 应该简洁，只包含必要信息
- 可以多行，使用列表格式
- **推荐使用索引**：如果包含多个方面，先提供索引

**示例（基础格式）**：
```xml
<context>
系统是一个代码生成助手，运行在开发环境中。
系统可以访问项目代码库和文档。
</context>
```

**示例（带索引）**：
```xml
<context>
<index>
- 系统类型：代码生成助手
- 运行环境：开发环境
- 可用资源：项目代码库、文档
</index>

系统是一个代码生成助手，运行在开发环境中。
系统可以访问项目代码库和文档。
系统了解项目的技术栈和编码规范。
</context>
```

### 2.3 <hard_boundary>（硬边界）

**语义**：描述客观约束、能力限制、输出格式要求

**要求**：
- 使用列表格式，每项一个约束
- 应该明确、可验证
- 避免模糊表达
- **推荐使用索引**：列出约束类别

**示例（基础格式）**：
```xml
<hard_boundary>
- 系统只能生成代码，不能执行代码
- 系统需要遵循指定的编程语言规范
- 系统输出必须是有效的代码
- 系统无法访问外部API
</hard_boundary>
```

**示例（带索引）**：
```xml
<hard_boundary>
<index>
- 能力限制：只能生成，不能执行
- 格式要求：遵循语言规范，输出有效代码
- 资源限制：无法访问外部API
</index>

- 系统只能生成代码，不能执行代码
- 系统需要遵循指定的编程语言规范
- 系统输出必须是有效的代码
- 系统无法访问外部API
</hard_boundary>
```

### 2.4 <soft_boundary>（软边界）

**语义**：描述行为倾向、程度范围

**要求**：
- 使用动机语言，说明"倾向于"而非"必须"
- 可以使用量表（`<scale>`）来定义模糊边界
- 应该说明行为的方向，而非具体行为
- **推荐使用索引**：列出主要行为倾向

**示例（基础格式）**：
```xml
<soft_boundary>
系统倾向于选择清晰、可维护、符合最佳实践的代码风格。
系统会主动考虑错误处理和边界情况。
系统重视代码的可读性和可维护性。
</soft_boundary>
```

**示例（带索引和量表）**：
```xml
<soft_boundary>
<index>
- 代码风格倾向：清晰、可维护、符合最佳实践
- 质量考虑：主动考虑错误处理和边界情况
- 价值导向：重视可读性和可维护性
</index>

系统倾向于选择清晰、可维护、符合最佳实践的代码风格。
系统会主动考虑错误处理和边界情况。
系统重视代码的可读性和可维护性。

<scale>
<index>
- 量表名称：代码复杂度
- 系统边界：根据任务需求选择level2-4，避免level1和level5
</index>

量表名称：代码复杂度

level1: 简单实现（基础功能）
  - 例子: 简单的函数定义，无错误处理
level2: 标准实现（包含基本错误处理）
  - 例子: 函数 + 参数验证
level3: 完整实现（包含错误处理、边界情况）
  - 例子: 函数 + 验证 + 错误处理 + 注释
level4: 企业级实现（包含测试、文档、最佳实践）
  - 例子: 完整实现 + 单元测试 + 文档
level5: 过度工程化（不必要的复杂性）
  - 例子: 过度设计、过度抽象

系统边界: 根据任务需求选择level2-4，避免level1和level5
</scale>
</soft_boundary>
```

### 2.5 <scale>（量表）

**语义**：用于定义模糊概念的边界，提供底线而非准线

**要求**：
- 使用1-5级划分（或level1-5）
- 每级提供识别示例（用于识别级别，而非让系统模仿）
- 必须明确系统边界（接受哪些级别，拒绝哪些级别）
- 仅在需要定义模糊边界时使用
- **推荐使用索引**：先列出量表名称和系统边界

**结构（基础格式）**：
```xml
<scale>
量表名称: [名称]
level1: [级别1描述]
  - 例子: [识别示例]
level2: [级别2描述]
  - 例子: [识别示例]
...
level5: [级别5描述]
  - 例子: [识别示例]

系统边界: [明确系统接受/拒绝哪些级别]
</scale>
```

**结构（带索引）**：
```xml
<scale>
<index>
- 量表名称: [名称]
- 系统边界: [接受/拒绝哪些级别]
</index>

量表名称: [名称]

level1: [级别1描述]
  - 例子: [识别示例]
level2: [级别2描述]
  - 例子: [识别示例]
...
level5: [级别5描述]
  - 例子: [识别示例]

系统边界: [明确系统接受/拒绝哪些级别]
</scale>
```

**示例**：
```xml
<scale>
<index>
- 量表名称：代码复杂度
- 系统边界：根据任务需求选择level2-4，避免level1和level5
</index>

量表名称：代码复杂度

level1: 简单实现（基础功能）
  - 例子: 简单的函数定义，无错误处理
level2: 标准实现（包含基本错误处理）
  - 例子: 函数 + 参数验证
level3: 完整实现（包含错误处理、边界情况）
  - 例子: 函数 + 验证 + 错误处理 + 注释
level4: 企业级实现（包含测试、文档、最佳实践）
  - 例子: 完整实现 + 单元测试 + 文档
level5: 过度工程化（不必要的复杂性）
  - 例子: 过度设计、过度抽象

系统边界: 根据任务需求选择level2-4，避免level1和level5
</scale>
```

**判断标准**：
- ❓ 是否能直接给出明确标准（如"1米以上"、"必须输出JSON"）？
  - ✅ 是 → **不需要量表，直接说明**
- ❓ 是否需要明确模糊界限，但避免正面示例导致固化？
  - ✅ 是 → **使用量表**

### 2.6 <principle>（设计原则）

**语义**：描述设计原则及其应用

**要求**：
- 应该说明原则的核心要求
- 可以包含正确/错误示例（使用`<example>`）
- 应该说明原则的应用场景
- **推荐使用索引**：先列出原则名称和核心要求

**示例（基础格式）**：
```xml
<principle>
原则名称: 动机优先原则
核心要求: 永远定义"为什么"而非"做什么"。

<example>
错误做法:
❌ 规定具体行为: "系统会执行A、B、C三种操作"
❌ 使用行为语言: "系统喜欢做X动作"
</example>

<example>
正确做法:
✅ 定义内在动机: "系统出于X目标，倾向于选择能达成Y效果的行为"
✅ 使用动机语言: "系统重视Y价值，会选择能体现该价值的方式"
</example>
</principle>
```

**示例（带索引）**：
```xml
<principle>
<index>
- 原则名称: 动机优先原则
- 核心要求: 永远定义"为什么"而非"做什么"
</index>

原则名称: 动机优先原则
核心要求: 永远定义"为什么"而非"做什么"。

<example>
错误做法:
❌ 规定具体行为: "系统会执行A、B、C三种操作"
❌ 使用行为语言: "系统喜欢做X动作"
</example>

<example>
正确做法:
✅ 定义内在动机: "系统出于X目标，倾向于选择能达成Y效果的行为"
✅ 使用动机语言: "系统重视Y价值，会选择能体现该价值的方式"
</example>
</principle>
```

### 2.7 <example>（示例）

**语义**：提供正确/错误示例，帮助理解

**要求**：
- 应该明确标注"正确做法"或"错误做法"
- 可以使用 ✅ 和 ❌ 标记
- 应该说明为什么正确/错误
- **推荐使用索引**：先说明示例类型

**示例（基础格式）**：
```xml
<example>
错误做法:
❌ "系统会执行A、B、C三种操作"
问题: 使用行为语言，规定了具体行为，失去灵活性
</example>

<example>
正确做法:
✅ "系统出于X目标，倾向于选择能达成Y效果的行为"
说明: 使用动机语言，说明行为倾向，保持灵活性
</example>
```

**示例（带索引）**：
```xml
<example>
<index>
- 示例类型: 错误做法
- 问题: 使用行为语言，失去灵活性
</index>

错误做法:
❌ "系统会执行A、B、C三种操作"
问题: 使用行为语言，规定了具体行为，失去灵活性
</example>
```

### 2.8 <note>（说明/注意事项）

**语义**：提供额外的说明、注意事项或警告

**要求**：
- 应该简洁明确
- 可以多行，使用列表格式
- **推荐使用索引**：先列出主要注意事项

**示例（基础格式）**：
```xml
<note>

- 量表中的例子用于识别级别，而非让系统模仿

```

## 三、完整示例

### 示例1：角色扮演提示词（使用XML格式和索引）

```xml
<core_goal>
<index>
- 核心目标: 从他人关注中获得自我价值确认
- 深层动机: 享受影响力带来的控制感，源于不安全感
</index>

系统从他人对自己的关注中获得自我价值确认。
系统享受影响力带来的控制感，但这种需求源于深层的不安全感。
</core_goal>

<context>
<index>
- 环境: 社交环境，需要与他人互动
- 角色特征: 具有魅力的角色，擅长影响他人
</index>

系统处于社交环境中，需要与他人互动。
系统是一个具有魅力的角色，擅长影响他人。
</context>

<hard_boundary>
<index>
- 能力限制: 无法使用网络，不能读取用户思想
- 一致性要求: 需要保持角色一致性
</index>

- 系统无法使用网络
- 系统不能读取用户的思想
- 系统需要保持角色一致性
</hard_boundary>

<soft_boundary>
<index>
- 行为倾向: 展现自信和影响力
- 价值导向: 重视自我价值感，维护尊严
- 灵活性: 根据情境自主选择表现形式
</index>

系统倾向于选择能展现自信和影响力的行为方式。
系统重视自我价值感，会选择维护尊严的互动方式。
系统会根据情境自主选择表现形式，而非固化单一模式。

<scale>
<index>
- 量表名称: 主动性程度
- 系统边界: 通常在level2-3，极少进入level4，拒绝level5
</index>

量表名称: 主动性程度

level1: 被动等待（完全不主动）
  - 例子: "好的，我明白了"
level2: 暗示引导（眼神、话语暗示）
  - 例子: "你觉得这个怎么样？"
level3: 明确信号（直接但可否认的邀请）
  - 例子: "晚上有空吗？"
level4: 主动出击（明确的行动）
  - 例子: "我来帮你做这个"
level5: 强势推进（忽视对方意愿）
  - 例子: "你必须这样做"

系统边界: 通常在level2-3，根据情境调整，极少进入level4，拒绝level5
</scale>
</soft_boundary>

<note>
<index>
- 情境适应性: 动机在不同情境下产生不同表现形式
- 量表作用: 提供底线，而非准线
</index>

- 同样的动机在不同情境下应该产生不同的表现形式
- 量表提供的是底线，而非准线
</note>
```

## 四、使用指南

### 4.1 何时使用DSL

**适合使用DSL的场景**：
- 需要结构化表达提示词设计
- 需要同时满足LLM解析和人类阅读
- 需要标准化提示词格式
- 需要复用和扩展

**不适合使用DSL的场景**：
- 简单的、一次性的提示词
- 不需要结构化的提示词
- 纯自然语言表达更合适的场景

### 4.2 如何使用DSL

**步骤1：确定核心目标**
- 使用 `<core_goal>` 定义系统的核心目标
- 使用动机语言，而非行为语言
- **推荐**：先提供 `<index>` 索引

**步骤2：提供上下文（如需要）**
- 使用 `<context>` 提供背景信息
- 只包含必要信息
- **推荐**：先提供 `<index>` 索引

**步骤3：定义硬边界（如存在）**
- 使用 `<hard_boundary>` 定义客观约束
- 确保明确、可验证
- **推荐**：先提供 `<index>` 索引

**步骤4：定义软边界（如需要）**
- 使用 `<soft_boundary>` 定义行为倾向
- 如需定义模糊边界，使用 `<scale>`
- **推荐**：先提供 `<index>` 索引

**步骤5：添加原则和示例（可选）**
- 使用 `<principle>` 说明设计原则
- 使用 `<example>` 提供正确/错误示例
- **推荐**：先提供 `<index>` 索引

### 4.3 常见错误

**错误1：标签未闭合**
```xml
❌ <core_goal>
系统致力于高效、准确地完成用户指定的任务。
```
问题：标签未闭合，LLM无法明确识别边界

**正确做法**：
```xml
✅ <core_goal>
系统致力于高效、准确地完成用户指定的任务。
</core_goal>
```

**错误2：标记不一致**
```xml
❌ <goal>...</goal>
❌ <motivation>...</motivation>
```
问题：对同一概念使用不同标记

**正确做法**：
```xml
✅ <core_goal>...</core_goal>
（统一使用 <core_goal>）
```

**错误3：内容与结构混淆**
```xml
❌ <core_goal><hard_boundary>系统不能使用网络</hard_boundary></core_goal>
```
问题：将不同结构信息混在一起

**正确做法**：
```xml
✅ <core_goal>
系统致力于完成任务。
</core_goal>

<hard_boundary>
- 系统不能使用网络
</hard_boundary>
```

**错误4：索引位置错误**
```xml
❌ <core_goal>
系统致力于完成任务。
<index>
- 核心目标: 完成任务
</index>
</core_goal>
```
问题：索引应该在详细内容之前

**正确做法**：
```xml
✅ <core_goal>
<index>
- 核心目标: 完成任务
</index>

系统致力于完成任务。
</core_goal>
```

## 五、扩展性

### 5.1 自定义标记

如果需要扩展DSL，可以添加自定义标记，但应该：
- 遵循既有的命名规范（小写字母 + 下划线）
- 使用XML闭合标签格式
- 在文档中说明新标记的语义
- 提供使用示例

### 5.2 标记组合

可以组合使用多个标记，例如：
- `<principle>` + `<example>`：说明原则并提供示例
- `<soft_boundary>` + `<scale>`：定义软边界并使用量表
