# STAGE ANALYZE: 系统性根本原因分析

## 五问法分析

### 问题：代码复杂，维护困难，增改复杂

#### 第一问：为什么代码复杂？
**答案**：GameView.vue 承担了过多职责，包含了 UI 渲染、状态管理、事件处理、业务逻辑等多个层面的代码。

#### 第二问：为什么 GameView 承担了过多职责？
**答案**：缺少合理的关注点分离（Separation of Concerns）。所有功能都直接写在主组件中，没有将相关逻辑提取到独立的 composables 或子组件中。

#### 第三问：为什么缺少关注点分离？
**答案**：在开发过程中，为了快速实现功能，采用了"先实现后优化"的方式，导致代码逐渐累积而没有及时重构。同时，缺少明确的架构设计指导原则。

#### 第四问：为什么没有及时重构？
**答案**：缺少代码审查机制和重构意识。当发现重复代码时，采用了复制粘贴的方式快速解决，而不是抽象提取。

#### 第五问（系统性根本原因）：为什么会出现这种开发模式？
**答案**：**缺少清晰的架构设计和模块化规范**。项目没有建立：
1. 明确的职责边界（哪些逻辑应该在 composables，哪些在 components）
2. DRY 原则的执行机制（如何识别和消除重复）
3. 组件设计规范（单一职责、开闭原则等）
4. 代码审查和重构流程

## 系统性根本原因

**核心问题**：架构设计缺失，导致代码组织混乱

### 具体表现：

1. **职责边界不清晰**
   - 组件、composables、utils 之间的职责划分不明确
   - 业务逻辑、UI 逻辑、状态管理混在一起

2. **缺少抽象层**
   - 时间输入逻辑应该抽象为独立的 composable
   - 行显示逻辑应该抽象为独立的 composable
   - 行渲染应该抽象为独立的组件

3. **缺少约定和规范**
   - 没有约定如何组织 composables
   - 没有约定如何设计组件接口
   - 没有约定如何处理状态管理

4. **技术债务累积**
   - 重复代码没有及时消除
   - 复杂逻辑没有及时拆分
   - 类型安全没有及时加强

## 假设验证

### 假设 1：如果提取时间输入逻辑到 composable，可以减少重复
**验证**：✅ 正确
- `handleTimeCharInput` 和 `handleTimeCharInputForChoice` 可以合并
- `handleTimePaste` 和 `handleTimePasteForChoice` 可以合并
- 时间字符串构建可以统一

### 假设 2：如果提取行显示逻辑到 composable，可以简化组件
**验证**：✅ 正确
- `currentLineIndex`, `displayedLines`, `typingRefs` 可以统一管理
- 行显示状态检查逻辑可以封装

### 假设 3：如果提取行渲染到组件，可以简化模板
**验证**：✅ 正确
- 每种行类型可以有自己的渲染组件
- 模板可以简化为 `<ScriptLineRenderer :line="line" />`

### 假设 4：如果建立清晰的架构规范，可以防止问题再次出现
**验证**：✅ 正确
- 明确的职责划分可以指导开发
- 代码审查可以基于规范进行
- 重构可以按规范执行

## 结论

**系统性根本原因**：**缺少清晰的架构设计和模块化规范，导致代码职责混乱、重复代码累积、维护困难。**

**影响范围**：
- 当前：GameView.vue 过于复杂
- 未来：如果不建立规范，新功能会继续加剧问题
- 团队：缺少规范会导致不同开发者采用不同模式

**解决方向**：
1. 建立清晰的架构规范
2. 按职责拆分代码到合适的层次
3. 消除重复代码
4. 加强类型安全
5. 建立代码审查机制
