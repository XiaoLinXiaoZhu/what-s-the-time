# 代码审查 - STAGE ANALYZE: 系统性根本原因分析

## 1. 五个为什么分析

### 为什么 #1: 为什么需要直接修改行对象的 status？

**表面原因**: 需要在运行时标记行已完成，防止重复操作。

**深层原因**: 
- `ScriptLine` 类型定义中 `status` 是可选的，但业务逻辑需要它
- 没有统一的状态管理机制来追踪行的生命周期状态
- 状态分散在数据对象本身，而不是独立的状态管理系统中

### 为什么 #2: 为什么 status 在类型定义中是可选的？

**表面原因**: 初始数据（ScriptSegment.lines）中不包含 status。

**深层原因**:
- **类型系统设计缺陷** - 混淆了"数据模型"和"运行时状态"
- 没有区分"静态剧本数据"和"动态运行时状态"
- 试图在同一个类型中表达两种不同的概念

### 为什么 #3: 为什么需要多个地方修改 displayedLines？

**表面原因**: 不同的交互（选择、输入、命令）需要不同的处理逻辑。

**深层原因**:
- **缺乏统一的状态变更抽象** - 没有状态机或统一的更新接口
- 每个功能点都需要直接操作底层数据结构
- 没有"命令模式"或"事件驱动"的架构

### 为什么 #4: 为什么需要手动同步状态（nextTick + setTimeout）？

**表面原因**: 需要等待 DOM 更新后再启动打字效果。

**深层原因**:
- **副作用管理缺乏抽象** - 副作用（打字效果、聚焦）需要手动触发
- 没有响应式的副作用系统
- 视图更新和业务逻辑耦合

### 为什么 #5: 为什么架构设计时没有考虑这些问题？

**系统性根本原因**: 
- **缺乏清晰的分层架构** - 数据层、状态层、视图层混合
- **没有状态管理策略** - 缺乏统一的状态管理抽象（如状态机、事件总线）
- **类型系统设计不完整** - 没有区分静态数据和运行时状态
- **职责划分不明确** - 组件、composables、工具函数职责重叠

## 2. 分层假设分析

### 假设 #1: Local Level（局部问题）
**假设**: 只是某些函数使用了 `as any`，修复这些函数即可。

**验证**: ❌ 不成立
- 问题遍布整个代码库
- 类型系统设计本身有问题
- 修复局部问题无法解决根本问题

### 假设 #2: Component Level（组件级问题）
**假设**: 问题在于组件职责不清，重构组件即可。

**验证**: ⚠️ 部分成立
- 组件确实职责不清
- 但更深层的问题是状态管理架构
- 仅重构组件无法解决状态管理问题

### 假设 #3: Systemic Level（系统性问题）✅
**假设**: 架构缺乏清晰的分层和状态管理抽象，需要重新设计架构。

**验证**: ✅ **成立**

**证据**:
1. **数据模型与运行时状态混淆**
   - `ScriptLine` 既表示静态数据，又承载运行时状态
   - 类型系统无法表达这种差异

2. **状态管理缺乏统一抽象**
   - 游戏状态、显示状态、行状态分散管理
   - 没有单一数据源（Single Source of Truth）
   - 状态变更没有统一接口

3. **副作用管理混乱**
   - 打字效果、聚焦等副作用需要手动触发
   - 没有响应式的副作用系统
   - 依赖时序（nextTick + setTimeout）

4. **数据流不清晰**
   - 多个地方可以修改同一状态
   - 难以追踪状态变更
   - 没有明确的数据流向

## 3. 系统性根本原因

### 根本原因 #1: 数据模型与运行时状态混淆

**问题**: 
- `ScriptLine` 类型试图同时表达静态剧本数据和动态运行时状态
- `status` 属性在类型定义中是可选的，但业务逻辑需要它

**影响**:
- 类型系统无法提供安全保障
- 需要使用 `as any` 绕过类型检查
- 数据模型和状态管理耦合

### 根本原因 #2: 缺乏统一的状态管理抽象

**问题**:
- 游戏状态、显示状态、行状态分散在多个 composables 中
- 没有单一数据源
- 状态变更没有统一接口
- 缺乏状态机或事件驱动架构

**影响**:
- 状态同步困难
- 多个地方可以修改同一状态
- 难以追踪状态变更
- 容易出现竞态条件

### 根本原因 #3: 副作用管理缺乏抽象

**问题**:
- 打字效果、聚焦等副作用需要手动触发
- 依赖时序（nextTick + setTimeout）
- 副作用逻辑分散在多个地方

**影响**:
- 代码重复
- 容易产生时序问题
- 难以测试
- 难以维护

### 根本原因 #4: 职责边界模糊

**问题**:
- 组件、composables、工具函数职责重叠
- `GameView` 既是视图又是控制器
- 业务逻辑分散在多个地方

**影响**:
- 难以理解和维护
- 难以测试
- 难以扩展

## 4. 架构缺陷总结

### 当前架构的问题

```
┌─────────────────────────────────────────┐
│         GameView (视图 + 控制器)         │
│  - 协调多个 composables                  │
│  - 手动同步状态                          │
│  - 处理副作用                            │
└─────────────────────────────────────────┘
           │
           ├─→ useGameState (全局状态)
           ├─→ useGameNavigation (导航逻辑)
           ├─→ useScriptDisplay (显示状态)
           ├─→ useLineState (行状态 - 未使用)
           └─→ useKeyboardNavigation (键盘)
           
问题:
- 状态分散，没有单一数据源
- 状态变更没有统一接口
- 副作用需要手动触发
- 类型系统被绕过
```

### 需要的架构

```
┌─────────────────────────────────────────┐
│         状态管理层 (State Layer)         │
│  - 统一的状态管理抽象                    │
│  - 单一数据源                            │
│  - 状态机/事件驱动                       │
└─────────────────────────────────────────┘
           │
           ├─→ 游戏状态 (GameState)
           ├─→ 显示状态 (DisplayState)
           └─→ 行状态 (LineState)
           
┌─────────────────────────────────────────┐
│         业务逻辑层 (Logic Layer)        │
│  - 纯函数处理业务逻辑                    │
│  - 通过状态管理层更新状态                │
│  - 返回副作用描述                        │
└─────────────────────────────────────────┘
           │
           ├─→ 导航逻辑
           ├─→ 选择处理
           └─→ 输入处理
           
┌─────────────────────────────────────────┐
│         视图层 (View Layer)             │
│  - 响应式渲染                            │
│  - 副作用执行器                          │
│  - 用户交互处理                          │
└─────────────────────────────────────────┘
```

## 5. 结论

**系统性根本原因**: 

**架构缺乏清晰的分层和统一的状态管理抽象**，导致：
1. 数据模型与运行时状态混淆
2. 状态管理分散，没有单一数据源
3. 副作用管理缺乏抽象
4. 职责边界模糊

**这导致需要大量补丁来**:
- 绕过类型系统（`as any`）
- 手动同步状态（watch + nextTick）
- 直接修改数据结构（splice, status 赋值）
- 手动触发副作用（startTyping）

**解决方案方向**:
1. **分离数据模型和运行时状态** - 创建独立的状态管理系统
2. **统一的状态管理抽象** - 单一数据源，统一更新接口
3. **副作用管理系统** - 响应式的副作用执行
4. **清晰的分层架构** - 数据层、状态层、逻辑层、视图层分离
