# 分析阶段报告：中间格式设计与文档拆分方案

## 一、系统性根因分析

### 问题诊断（五问法）

**问题1：为什么编剧需要学习技术标签？**
- 因为当前文档将创作指导和技术实现混合在一起

**问题2：为什么文档会混合创作和技术内容？**
- 因为缺乏清晰的职责分离和中间格式

**问题3：为什么缺乏中间格式？**
- 因为文档设计时没有考虑编剧和技术美术的工作流程差异

**问题4：为什么没有考虑工作流程差异？**
- 因为文档是单一视角编写的，没有区分用户角色

**问题5（系统性根因）：为什么文档是单一视角？**
- **系统性根因**：文档设计时缺乏**角色分离**和**格式抽象层**的设计思维

### 系统性解决方案

需要建立**三层架构**：
1. **创作层**（编剧）：自然语言创作，使用语义化标记
2. **中间层**（格式桥接）：结构化但人类可读的格式
3. **实现层**（技术美术）：程序格式，包含所有技术细节

## 二、中间格式设计

### 设计原则

1. **自然性**：编剧可以像写小说一样创作
2. **语义化**：使用语义标记而非技术标记
3. **可转换性**：技术美术可以系统化转换
4. **可读性**：人类可读，便于审查和修改

### 中间格式规范（Markdown扩展）

#### 1. 文本类型标记（语义化）

**主角内心独白**（对应 `narration`）：
```markdown
[内心] 早晨的阳光透过梧桐叶片的缝隙，像某种精致的花边装饰。
这是新学校的第一周。一切都应该是完美的。
```

**主角说出口的话**（对应 `dialogue` 无 character）：
```markdown
[说] 她指着街角那家新开的包子店，眼睛亮晶晶的。"我们去试试那个吧？听说皮薄馅大，很香的。"
```

**其他角色对话**（对应 `dialogue` 有 character）：
```markdown
[B] 你刚才有没有看到？我们班那个H，刚才在走廊里……
```

#### 2. 节奏与停顿标记（语义化）

**自然停顿**（对应 `{delay:0.5}`）：
```markdown
[停顿:短] 对，一定是梦。
```

**句间呼吸**（对应 `{delay:0.8}`）：
```markdown
[停顿:中] 时间不对。
哪里不对？
```

**段落转折**（对应 `{delay:1.0}`）：
```markdown
[停顿:长] 我看着她的背影，想起了一个问题。
```

**换行**（对应 `{br}`）：
```markdown
时间不对。
哪里不对？
我试图抓住什么，但只有虚无。
```
（自然换行，无需标记）

#### 3. 视觉强调标记（语义化）

**时间信息**（对应 `{red}{typewriter}...{/typewriter}{/red}`）：
```markdown
[时间:10:37] 手表上的指针，停在了这一刻。
```

**关键念头**（对应 `{bold}...{/bold}`）：
```markdown
[强调] 时间不对。
```

**危险/执念**（对应 `{red}...{/red}`）：
```markdown
[危险] 不……停下……
```

**记忆模糊**（对应 `{blur}...{/blur}`）：
```markdown
时间是……[模糊]10:37[模糊]……应该吧……
```

**记忆修正**（对应 `{strike}...{/strike}`）：
```markdown
[修正:旧的想法] [强调:新的想法]
```

**逐字强调**（对应 `{typewriter}...{/typewriter}`）：
```markdown
[逐字] 她的名字
```

#### 4. 交互元素标记

**选择分支**：
```markdown
[选择]
- 上前为H辩护，维护朋友关系
  [内心] 我转过身，脸上带着恰到好处的关切。"你们在说什么？H怎么了？"
  [B] 哦，我们只是……没什么大不了的。
  [标记:FLAG_Appears_Defensive]

- 装作没听到，继续往前走
  [内心] 我继续往前走，假装什么也没听到。有些事情，知道得太多反而不好。
  [标记:FLAG_Ignores_Conflict]

- 收集更多关于H的信息
  [内心] 我假装在远处等人，刻意地停留了一段时间。她们的谈话声变得更加清晰。
  [标记:FLAG_Habit_Observed]
```

**时间输入**：
```markdown
[输入时间] 请输入时间（HH:MM格式）
```

**命令**：
```markdown
[命令:设置标记] FLAG_Game_Start
```

#### 5. 元数据标记

**片段信息**（在文件开头）：
```markdown
---
片段ID: P1-DTL-T0805-Breakfast
时间: 08:05
阶段: P1
解锁条件: FLAG_Game_Start
描述: 阶段1 - 早餐
作用: 建立"虚构恋爱史"的开端；情感绑定。
---
```

### 中间格式示例

完整示例：
```markdown
---
片段ID: P1-DTL-T0805-Breakfast
时间: 08:05
阶段: P1
解锁条件: FLAG_Game_Start
描述: 阶段1 - 早餐
作用: 建立"虚构恋爱史"的开端；情感绑定。
---

[内心] 早晨的阳光透过梧桐叶片的缝隙，像某种精致的花边装饰。
[停顿:中] 这是新学校的第一周。一切都应该是完美的。

[说] 她指着街角那家新开的包子店，眼睛亮晶晶的。"我们去试试那个吧？听说皮薄馅大，很香的。"

[说] 我看着她，心里涌起一种奇怪的情绪。不是爱——那种东西太虚假了。
[停顿:中] 是一种……占有欲？不对。是保护欲？也不对。
[停顿:中] "你的胃不好，早上吃这么油腻的东西会不舒服的。"

[B] 她眨了眨眼，像是被什么小东西绊了一下。"可是……偶尔试试也没关系吧？"

[说] "偶尔？"我笑了笑，去旁边的便利店买了三明治。
[停顿:中] "你看，这个蛋白含量高，营养更均衡。"
[停顿:中] 她接过三明治，咬了一口，然后露出一个标准化的笑容。"嗯，很好吃。"

[内心] 那种笑容。
[停顿:中] 我见过。在很久很久以前，另一个女孩也露出过这样的笑容。
[停顿:中] 那种被训练出来的、迎合式的笑容。

[说] "中午的时候，我们还是去那个老地方吧。"她走在前面，声音里没有任何起伏。
[停顿:中] "还是老地方。"我重复着，仿佛在确认什么。

[内心] 我看着她的背影，想起了一个问题。
[停顿:长] 她今天露出的疲惫笑容，是从哪里学来的？
```

## 三、文档拆分方案

### 拆分原则

1. **职责分离**：编剧专注创作，技术美术专注转换
2. **信息完整**：两份文档必须包含所有必要信息
3. **交叉引用**：必要时引用另一份文档
4. **独立可用**：每份文档可以独立使用

### 文档结构设计

#### 文档A：编剧创作指南（`编剧创作指南.md`）

**目标读者**：编剧
**核心内容**：
1. 核心理念（文风目标、情感阶段、四大支柱）
2. 写作原则（四大支柱的执行标准）
3. 表现技法（反复/纠结、比喻、精简）
4. 中间格式规范（如何使用语义化标记创作）
5. 创作示例和最佳实践

**不包含**：
- 技术标签细节（`{br}`, `{delay:0.8}`等）
- 程序结构细节（TypeScript类型定义）
- 转换规则（这些在技术美术文档中）

#### 文档B：技术美术转换指南（`技术美术转换指南.md`）

**目标读者**：技术美术
**核心内容**：
1. 中间格式规范（理解编剧使用的标记）
2. 转换规则（从中间格式到程序格式的映射）
3. 标签系统详解（所有技术标签的使用规范）
4. 文本类型区分（dialogue vs narration的判断标准）
5. 交互设计规范（choice、timeChoice等的实现）
6. 转换示例和工具建议

**不包含**：
- 文风原则的详细解释（这些在编剧文档中）
- 创作技法（这些是编剧的职责）

### 交叉引用策略

1. **编剧文档**中引用技术美术文档：
   - "关于技术标签的具体实现，请参考《技术美术转换指南》"
   - "如需了解程序格式细节，请咨询技术美术"

2. **技术美术文档**中引用编剧文档：
   - "关于文风原则的详细说明，请参考《编剧创作指南》"
   - "如需理解创作意图，请参考编剧文档中的表现技法部分"

## 四、转换规则设计

### 映射表（中间格式 → 程序格式）

| 中间格式 | 程序格式 | 说明 |
|---------|---------|------|
| `[内心] ...` | `{ type: 'narration', text: '...' }` | 主角内心独白 |
| `[说] ...` | `{ type: 'dialogue', text: '...' }` | 主角说出口的话 |
| `[角色名] ...` | `{ type: 'dialogue', character: '角色名', text: '...' }` | 其他角色对话 |
| `[停顿:短]` | `{delay:0.5}` | 自然停顿 |
| `[停顿:中]` | `{delay:0.8}` | 句间呼吸 |
| `[停顿:长]` | `{delay:1.0}` | 段落转折 |
| `[时间:HH:MM]` | `{red}{typewriter}HH:MM{/typewriter}{/red}` | 时间信息 |
| `[强调] ...` | `{bold}...{/bold}` | 关键念头 |
| `[危险] ...` | `{red}...{/red}` | 危险/执念 |
| `[模糊]...[模糊]` | `{blur}...{/blur}` | 记忆模糊 |
| `[修正:旧] [强调:新]` | `{strike}旧{/strike} {red}新{/red}` | 记忆修正 |
| `[逐字] ...` | `{typewriter}...{/typewriter}` | 逐字强调 |
| `[选择] ...` | `{ type: 'choice', choices: [...] }` | 选择分支 |
| `[标记:FLAG]` | `{ type: 'command', command: 'setFlag', params: { flag: 'FLAG' } }` | 设置标记 |

### 转换流程

1. **解析中间格式**：读取Markdown文件，解析语义标记
2. **应用转换规则**：根据映射表转换为程序格式
3. **生成TypeScript代码**：输出符合 `ScriptSegment` 接口的代码
4. **验证和优化**：检查格式正确性，优化标签使用

## 五、验证标准

### 中间格式验证
- ✅ 编剧可以像写小说一样自然创作
- ✅ 所有语义标记清晰易懂
- ✅ 格式完整，包含所有必要信息

### 文档拆分验证
- ✅ 编剧文档包含所有创作指导
- ✅ 技术美术文档包含所有转换规则
- ✅ 两份文档可以独立使用
- ✅ 交叉引用清晰有效

### 转换规则验证
- ✅ 所有中间格式标记都有对应的程序格式
- ✅ 转换规则无歧义
- ✅ 转换后的代码符合程序规范
