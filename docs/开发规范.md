## AI 时代的游戏工程重构：走向闭环与状态驱动

在传统的游戏开发模式中，逻辑与表现往往高度耦合。这种“开环系统”极大地增加了自动化测试的难度。**在 AI 辅助开发的时代，必须建立可计算的反馈回路（闭环系统）。** 只有剥离出纯粹的数学模型，AI 才能通过单元测试稳定迭代代码。

本指南旨在提供一套基于 Godot (.NET) 的架构重构方案，将 Web 领域的 MVVM 与状态驱动思想引入游戏开发。

### 核心理念：表现只是状态的副作用

游戏本质上是一个随时间步进的状态机。**画面的渲染、声音的播放，仅仅是状态机在当前帧的副作用。** 重构的核心目标是：业务逻辑不需要引擎 API 即可独立运行，引擎节点只负责监听数据并更新画面。

### 模块职责划分

为了实现彻底的解耦，必须严格界定各个模块的归属与处理方式。

| 模块分类 | 职责描述 | 核心特征 |
| :--- | :--- | :--- |
| **核心领域 (Domain)** | 处理数值计算、规则判定与核心状态。 | 纯 C# 代码，**无 Godot 依赖**，完全可单元测试。 |
| **状态容器 (State)** | 存储当前游戏数据，提供事件订阅接口。 | 逻辑与表现的桥梁，单向数据流。 |
| **外部输入 (Input)** | 捕获玩家操作，转化为命令。 | 仅发送意图，不直接修改状态。 |
| **外部服务 (Service)** | 提供物理碰撞、网络通信等底层能力。 | 通过接口抽象（依赖反转），逻辑层不直接调用引擎。 |
| **视觉表现 (View)** | 监听状态变化，播放动画或音效。 | **哑视图（Dumb View）**，无任何判断逻辑。 |

### Godot (.NET) 重构实操指南

#### 1. 构建纯净的领域层

将核心逻辑从 Godot 节点中剥离。新建独立的 C# 类或类库，**严禁引入 `Godot` 命名空间。**

```csharp
// 纯净的 C# 类
public class PlayerModel {
    public int Health { get; private set; }
    public event Action<int> OnHealthChanged;
    public event Action OnDied;

    public PlayerModel(int initialHealth) {
        Health = initialHealth;
    }

    public void TakeDamage(int damage) {
        if (Health <= 0) return;
        Health -= damage;
        OnHealthChanged?.Invoke(Health);
        if (Health <= 0) OnDied?.Invoke();
    }
}
```

#### 2. 将节点降级为“哑视图”

Godot 的 `Node` 脚本不再负责业务计算。它的职责被缩减为：**绑定数据、渲染状态、传递输入。**

```csharp
using Godot;
using System;

public partial class PlayerView : Node2D {
    private PlayerModel _model;
    [Export] private ProgressBar _healthBar;

    public void Setup(PlayerModel model) {
        _model = model;
        // 订阅核心逻辑的事件
        _model.OnHealthChanged += UpdateHealthUI;
        _model.OnDied += PlayDeathAnimation;
    }

    private void UpdateHealthUI(int newHealth) {
        _healthBar.Value = newHealth;
    }

    private void PlayDeathAnimation() {
        // 仅处理视觉副作用
        QueueFree();
    }
}
```

#### 3. 隔离并反转物理服务

游戏通常依赖引擎的物理系统。为了保持逻辑层的纯洁性，必须使用**接口（Interface）**进行依赖反转。

*   **逻辑层定义接口：** `public interface IPhysicsService { bool CheckCollision(Vector2 pos); }`
*   **引擎层实现接口：** 由 Godot 的 `PhysicsServer2D` 具体实现该接口。
*   **测试层模拟接口：** 在单元测试中 Mock 一个假的物理服务，无需启动引擎即可测试移动逻辑。

### 测试闭环的验证

完成剥离后，工程将具备极强的可测试性。**只要纯逻辑层的单元测试通过，游戏的核心规则即被证明是可靠的。**

```csharp
[Fact]
public void Player_TakeLethalDamage_TriggersDeathEvent() {
    var player = new PlayerModel(100);
    bool isDead = false;
    player.OnDied += () => isDead = true;

    player.TakeDamage(100);

    Assert.True(isDead);
    Assert.Equal(0, player.Health);
}
```

如果在游戏中发现表现异常（例如玩家未播放死亡动画），这不再是逻辑 Bug，而是 Godot 视图层的数据绑定遗漏或美术资源配置错误。这种架构将复杂的系统问题降维成了简单的表现层配置问题。**这为 AI 自动化生成代码并自行验证提供了坚实的工程基础。**
