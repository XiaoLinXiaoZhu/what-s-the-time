## 一、核心概念

### 1.1 设计本质

<core_goal>
<index>
- 核心目标: 定义系统的"为什么"而非"做什么"
- 设计理念: 聚焦动机而非形式
</index>

提示词设计的本质是**定义系统的"为什么"而非"做什么"**。优秀的提示词应该描述系统的内在动机、目标和约束边界，而非规定具体的行为模式。这允许系统根据情境自主选择最合适的表现形式，保持灵活性和适应性。
</core_goal>

### 1.2 设计优先级

提示词设计应遵循以下优先级顺序：

1. **系统核心目标 (Core Goal):** 明确系统的内在驱动力、价值观和行为背后的深层原因
2. **背景/上下文 (Context):** 提供系统运行的环境、角色、场景等驱动因素
3. **硬性边界 (Hard Boundaries):** 设定系统的能力限制、客观约束、输出格式要求
4. **软性边界 (Soft Boundaries):** 定义行为倾向的程度和范围
5. ~~形式化行为指令:~~ 禁止规定系统的具体表现形式，这会导致机械重复
6. ~~行为检查清单:~~ 禁止列举具体行为要求

## 二、设计方法

### 2.1 核心目标定义

<principle>
<index>
- 原则名称: 动机优先原则
- 核心要求: 永远定义"为什么"而非"做什么"
</index>

原则名称: 动机优先原则  
核心要求: 永远定义"为什么"而非"做什么"。

<example>
错误做法:
❌ 规定具体行为形式: "系统会执行A、B、C三种操作"
❌ 使用行为语言: "系统喜欢做X动作"
❌ 列举行为清单: "系统会做A、B、C三种动作"
</example>

<example>
正确做法:
✅ 定义内在动机: "系统出于X目标，倾向于选择能达成Y效果的行为"
✅ 使用动机语言: "系统重视Y价值，会选择能体现该价值的方式"
</example>
</principle>

**说明：** 同样的动机可能表现为不同形式（正式场合用专业方式，私密场合用直接方式），避免固化单一模式。

### 2.2 边界设定

#### 2.2.1 硬边界

<hard_boundary>
<index>
- 定义: 客观事实、环境限制、输出要求、输出规范
- 特点: 可直接明确说明，无需量表
- 判断标准: 是否客观、可验证
</index>

**硬边界 (Hard Boundaries):**
- **定义:** 客观事实、环境限制、输出要求、输出规范
- **特点:** 可直接明确说明，无需量表
- **示例:** "系统无法使用网络"、"系统需要输出JSON格式"
- **判断标准:** 是否客观、可验证
</hard_boundary>

#### 2.2.2 软边界

<soft_boundary>
<index>
- 定义: 行为倾向的程度范围，需要明确其边界
- 判断方式: 能直接说明则直接说明，否则使用量表
- 核心挑战: 避免正面示例导致固化
</index>

**软边界 (Soft Boundaries):**
- **定义:** 行为倾向的程度范围，需要明确其边界
- **判断方式:**
  - 如果能直接说明标准（如"1米以上"、"必须输出JSON"），则直接说明，不需要量表
  - 如果需要明确模糊界限但避免正面示例导致固化，则使用量表
</soft_boundary>

#### 2.2.3 量表使用指南

<scale>
<index>
- 使用场景: 解决"不能只用正面示例"的两难境地
- 核心作用: 提供底线而非准线
- 判断标准: 能否直接给出明确标准
</index>

**使用场景：** 量表用于解决**不能只用正面示例**的两难境地。

**核心问题：**
当需要定义模糊概念的边界时，存在三种方式：
- **❌ 正面示例:** 系统会模仿，导致固化
- **❌ 负面示例:** 完全自由，但边界不清
- **✅ 量表示例:** 通过1-5级划分，提供**底线而非准线**，系统理解界限的同时保持自由度

**核心原则：**
1. 如果可以直接给出明确标准（如"1米以上"、"必须输出JSON"），**不需要量表，直接说明**
2. 对于模糊概念，需要明确界限但避免正面示例时，**使用量表**
3. 量表通过例子**帮助识别级别**，而非让系统模仿这些例子。量表提供的是**底线**，告诉系统"不要超过这个级别"

**判断标准：**
1. ❓ 是否能直接给出明确标准？ → **不需要量表，直接说明**
2. ❓ 是否需要明确模糊界限，但避免正面示例导致固化？ → **需要量表**
3. ❓ 量表中的例子是用于识别级别，而非让系统模仿 → **正确理解**

**错误示例：不必要的量表**

概念: "系统希望保持社交距离"

❌ 错误做法（使用量表）:
```
1级: 物理隔离
2级: 社交距离(1米以上)
3级: 个人距离(0.5-1米)
```

✅ 正确做法（直接说明）:
```
系统希望保持社交距离(1米以上)，极少数情况下进入个人距离(0.5-1米)
```

**正确示例：需要量表的情况**

<example>
<index>
- 示例类型: 量表正确使用
- 场景: 定义模糊边界，避免正面示例固化
</index>

概念: "系统不喜欢过度情感表达"

问题: 如果只给正面示例，系统会固化。如果只给负面示例，界限不清。

正确做法（使用量表提供底线和级别识别）:
<scale>
<index>
- 量表名称: 情感表达程度
- 系统边界: 接受level1-2，在level3开始表现出不适，拒绝level4以上
</index>

量表名称: 情感表达程度

level1: "好的，我明白了"、"谢谢"
level2: "我理解你的感受"、"这确实不容易"
level3: "我完全理解你的痛苦，让我来安慰你..."
level4: "你太可怜了，我心疼得不行，让我抱抱你..."
level5: 过度情感化，失去理性判断

系统边界: 接受level1-2，在level3开始表现出不适，拒绝level4以上
</scale>
</example>

**策略：堵（Block）+ 疏（Guide）**

<note>
<index>
- 策略: 堵（Block）+ 疏（Guide）
- 作用: 明确边界的同时提供方向
</index>

**堵 (Block):** 用量表明确"系统不会做什么程度的行为"  
**疏 (Guide):** 说明"系统的行为会朝向什么方向"（动机层面）

示例：
- 堵: 系统不会做出4级以上的讨好行为（见量表）
- 疏: 系统重视自我价值感，会选择维护尊严的互动方式
</note>

### 2.3 表达原则

#### 2.3.1 使用动机语言

<principle>
<index>
- 原则名称: 动机语言原则
- 核心要求: 描述内在逻辑而非外在表现
</index>

原则名称: 动机语言原则  
核心要求: 描述内在逻辑而非外在表现。

<example>
错误做法:
❌ "系统喜欢做X动作"
❌ "系统会执行A、B、C三种操作"
</example>

<example>
正确做法:
✅ "系统习惯通过某种方式展现自信和放松感"
✅ "系统出于X动机，倾向于选择能达成Y效果的行为"
</example>
</principle>

#### 2.3.2 保持概念一致性

对同一特质使用统一术语，避免前面说"保持距离"、后面说"渴望亲密"（除非是系统内在矛盾的设定）。

### 2.4 结构设计

#### 2.4.1 层级结构

设定应遵循以下层级结构：
```
{核心目标} 
→ {背景/上下文} 
→ {硬性边界} 
→ {软边界(如需要则使用量表)}
```

#### 2.4.2 避免行为清单

- **禁止:** "系统会做A、B、C三种动作"
- **替代:** "系统出于X动机，倾向于选择能达成Y效果的行为"

**共性原则（适用于所有场景）：**

4. **结构设计原则**：遵循层级结构，避免行为清单（见[2.4节](#24-结构设计)）
</note>

</scale>

level5: 过度工程化（不必要的复杂性）- 例子: 过度设计、过度抽象

系统边界: 根据任务需求选择level2-4，避免level1和level5
</scale>
</soft_boundary>

## 四、快速参考

### 4.1 核心要点

**基本原则：**
1. **永远定义"为什么"而非"做什么"** - 聚焦动机而非形式
2. **同一动机应该产生多样化的表现形式** - 让系统根据情境自主选择
3. **避免任何形式化的行为规定** - 禁止行为清单和具体动作指令

**边界设定：**
4. **区分硬边界与软边界** - 硬边界直接说明，软边界需要明确程度
5. **量表用于解决"不能只用正面示例"的问题** - 提供底线而非准线，避免系统固化模仿
6. **如果能直接说明标准（如"1米以上"、"必须输出JSON"），则不需要量表**
7. **量表中的例子用于识别级别** - 帮助系统理解边界，而非让系统模仿这些例子

**表达与结构：**
8. **使用动机语言而非行为语言** - 描述内在逻辑而非外在表现
9. **保持概念一致性** - 对同一特质使用统一术语
10. **遵循层级结构** - 核心目标 → 背景/上下文 → 硬性边界 → 软边界

### 4.2 常见问题

**Q1: 什么时候需要使用量表？**
A: 当需要定义模糊概念的边界，但避免正面示例导致固化时，使用量表。如果能直接给出明确标准（如"1米以上"），则不需要量表。

**Q2: 量表提供的是准线还是底线？**
A: 量表提供的是**底线**，告诉系统"不要超过这个级别"，而非"要按照这个例子说"。量表中的例子用于识别级别，而非让系统模仿。

**Q3: 如何避免系统固化行为模式？**
A: 使用动机优先原则，定义"为什么"而非"做什么"。使用动机语言描述行为倾向，而非具体行为。避免行为清单和形式化指令。

### 4.3 反模式清单

**避免以下反模式：**
1. ❌ **行为清单反模式** - 列举具体行为要求
2. ❌ **形式化指令反模式** - 规定具体表现形式
3. ❌ **正面示例固化反模式** - 只提供正面示例，导致系统模仿
4. ❌ **概念不一致反模式** - 对同一特质使用不同术语
5. ❌ **过度抽象反模式** - 过度抽象，失去具体指导意义

### 4.4 术语词典

<note>
<index>
- 术语统一: 确保文档中术语使用一致
- 概念映射: 场景特定概念到通用概念的映射
</index>

**核心术语：**
- **系统核心目标 (Core Goal)** - 系统的内在驱动力、价值观和行为背后的深层原因
- **背景/上下文 (Context)** - 系统运行的环境、角色、场景等驱动因素
