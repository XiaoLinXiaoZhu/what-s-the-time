## 一、任务理解

用户要求设计一个**多步骤提示词系统**，实现以下目标：

1. **类似diffusion模型的渐进式转换**：将游戏的基础概念逐步细化，最终转换为具体的游戏剧情脚本

3. **可检查的中间日志**：每一步都产出可以快速校验的结果
4. **拆分优化、细化和校验步骤**：类似 `schedule → checker1 → detailer2 → etc`

- **输入**: 游戏基础概念（如片段元数据、情节描述、作用说明等）
- **输出**: 完整的游戏脚本（如 `P0-CORE-T1037-FallBlur.ts`）

### 2.2 现有转换流程

**当前流程**（手动转换）：
```
基础概念 → 编剧创作（中间格式） → 技术美术转换 → 最终脚本（TypeScript）
```

**存在的问题**：
1. **依赖人工理解**：编剧需要深刻理解文风原则和情节设计
2. **转换需要艺术判断**：技术美术需要理解创作意图，手动判断节奏和强调
3. **难以标准化**：每个环节都依赖创作者的个人理解
4. **无中间校验**：编剧创作到技术美术转换之间没有明确的校验步骤

### 2.3 示例对比分析

#### 输入示例（中间格式 - P0-CORE-T1037-FallBlur.md）

```markdown
---
片段ID: P0-CORE-T1037-FallBlur
时间: 10:37
阶段: P0
解锁条件: 
描述: 初始循环 - 坠落模糊
作用: 建立核心悬念；植入核心时间锚点。
---

*风声。*
*尖锐的、撕裂空气的风声。*
*从四面八方涌来，将我包裹。*
...
```

#### 输出示例（程序格式 - P0-CORE-T1037-FallBlur.ts）

```typescript
export const P0_CORE_T1037_FallBlur: ScriptSegment = {
  id: 'P0-CORE-T1037-FallBlur',
  time: '10:37',
  description: '初始循环 - 坠落模糊',
  loop: 'P0',
  unlockFlags: [],
  lines: [
    {
      type: 'narration',
      text: '风声。{br}{delay:0.8}尖锐的、{delay:0.5}撕裂空气的风声。...'
    },
    ...
  ]
}
```

#### 转换复杂度分析

**元数据转换**：
- 相对简单，机械映射：`片段ID` → `id`, `时间` → `time`

**文本转换**：
- **高度复杂**，需要理解：
  1. 文本类型判断（narration/dialogue）
  2. 节奏控制（何时使用`{br}{delay:X}`）
  3. 视觉强调（何时使用`{bold}`/`{red}`/`{typewriter}`）
  4. 情节理解（为什么这里需要停顿0.8秒而非0.5秒）

**关键发现**：
- 转换不是机械的，而是**理解创作意图**的艺术性转换
- 每个转换决策都基于对**文风原则**和**情节意图**的深刻理解

### 3.1 核心挑战识别

基于上述分析，我识别出以下核心挑战：

| 挑战 | 描述 | 影响 |
|-----|------|------|
| **C1: 创作意图的理解** | 需要深刻理解文风原则（四大支柱）和情节设计 | 影响整个创作质量 |
| **C2: 艺术判断的标准化** | 节奏、强调等艺术判断难以标准化 | 难以设计提示词 |
| **C3: 中间状态的可验证性** | 缺乏明确的中间检查点 | 无法快速发现问题 |
| **C4: 多层次知识的集成** | 需要同时理解游戏设计、文风、技术规范 | 提示词设计复杂 |

### 3.2 diffusion模型类比分析

用户提到"类似diffusion模型"，这给了我重要启发：

**diffusion模型的特点**：
1. **渐进式去噪**：从噪声逐步细化到清晰图像
2. **每步都有明确的数学操作**：加噪/去噪
3. **可控的中间状态**：每一步的输出都是可验证的
4. **多尺度信息**：从粗糙到精细

**应用到脚本生成**：
```
基础概念（噪声） 
  → 结构设计（粗糙轮廓）
  → 初稿撰写（基本形态）
  → 文风优化（细节打磨）
  → 格式转换（最终输出）
```
